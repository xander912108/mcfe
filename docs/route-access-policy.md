# Route Access Policy — Mentori Club

Документ фиксирует договорённость для шага 3.8A: как обсуждать и проектировать route guards без немедленного изменения текущего роутинга, `HashRouter`, живых страниц и демо-сценариев.

## Статус

- Это policy-документ, а не реализация guards.
- Текущий прямой доступ к route не меняется.
- Навигационная видимость уже может зависеть от роли и feature flags, но это не считается полноценной защитой route.
- Pure access helpers добавлены без подключения runtime redirects к router.
- Любой guard должен внедряться отдельным PR после согласования источника auth и fallback-сценариев.

## Цели

1. Защитить лидерские поверхности от случайного доступа в production-сценариях.
2. Не сломать текущие живые страницы и вложенные лидерские route.
3. Сохранить возможность демо/презентационного режима до подключения реального auth/store.
4. Сделать будущие redirects бережными и понятными для пользователя.
5. Не превращать Mentori Club в интерфейс запрета, контроля или наказания.

## Что уже есть сейчас

- `navigationConfig` описывает существующие route, роли, surfaces и bindings к текущим компонентам.
- `NavigationAccessProvider` централизует demo role и feature flags для навигации.
- `useNavigation` фильтрует пункты меню по `minRole` и feature flags.
- `App.tsx` сохраняет текущий shell, `HashRouter`-совместимые route bindings и mobile bottom nav.

Важно: скрытие пункта меню не равно запрету прямого перехода по URL.

## Роли и базовая иерархия

Текущая техническая иерархия ролей:

| Роль | Смысл для доступа |
| --- | --- |
| `guest` | Публичный посетитель без входа. |
| `newcomer` | Новый участник, которому доступен базовый стартовый опыт. |
| `member` | Участник сообщества с доступом к основным разделам. |
| `mentor` | Участник с расширенной ролью поддержки, но не владелец лидерской консоли. |
| `leader` | Лидер сообщества с доступом к консоли лидера. |

До подключения реального auth/store эта иерархия используется только для видимости навигации и проектирования будущих guards.

## Источник правды для доступа

Перед реализацией route guards нужно согласовать один источник правды:

1. Реальный auth/store приложения.
2. Backend/API session или profile endpoint.
3. Явный demo mode для презентационных сценариев.

Нельзя считать demo default `role = 'leader'` production-моделью доступа.

## Поведение при прямом переходе без доступа

Для Mentori Club предпочтителен бережный fallback, а не резкий “запрещено”.

### Рекомендуемый production-сценарий

Если пользователь без роли `leader` открывает лидерский route:

1. Не рендерить лидерскую консоль.
2. Показать мягкий экран доступа или сделать мягкий redirect на безопасную страницу.
3. Не использовать формулировки в стиле наказания, контроля или блокировки.
4. Дать понятное следующее действие: вернуться в сообщество, запросить доступ или открыть свой путь.

### Предпочтительный fallback

Приоритетный вариант для будущей реализации:

- для участника: мягкий redirect на `/my-path` или экран объяснения;
- для гостя: redirect/переход к auth flow, когда auth flow будет согласован;
- для demo mode: сохранить доступ, если demo role явно `leader`.

## Бережные формулировки fallback UI

Использовать:

- “Этот раздел доступен лидеру сообщества.”
- “Если вы помогаете команде или ведёте пространство, можно запросить доступ.”
- “Вернуться в сообщество.”
- “Открыть мой путь.”

Не использовать:

- “Доступ запрещён.”
- “Недостаточно прав.”
- “Ошибка доступа.”
- “Вы не администратор.”

## Feature flags и guards

Feature flags могут скрывать пункты навигации и в будущем ограничивать route, но только если это не ломает существующие сценарии.

Правило:

- выключенный feature flag скрывает entry point;
- прямой route закрывается только после отдельного решения;
- если route закрывается feature flag, fallback должен быть мягким и объясняющим.

## Demo mode

До подключения production auth нужен явный demo-сценарий.

Рекомендуемое правило:

- demo mode должен явно задавать роль и flags;
- demo mode не должен быть неявной production-логикой;
- если demo mode выключен, guards должны опираться на реальный источник доступа.

## Что запрещено делать в первом PR с guards

- Менять `HashRouter` на `BrowserRouter`.
- Схлопывать `/leader/*` до одного route.
- Заменять живые страницы заглушками.
- Удалять props-based bindings без отдельной миграции.
- Включать жёсткие redirects без согласованного fallback UX.
- Использовать демо-роль `leader` как production security model.
