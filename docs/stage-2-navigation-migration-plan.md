# План дальнейшей миграции после этапов 2A–2F

Документ описывает, как переводить текущую логику Mentori Club на новые навигационные рельсы без резкой смены интерфейса, потери страниц и ломки лидерской консоли.

## Главный принцип

Новые архитектурные слои уже добавлены, но текущий `App.tsx` остаётся рабочим shell приложения. Поэтому дальнейшая миграция должна идти маленькими PR:

- один PR — один ограниченный перенос;
- штатный интерфейс не должен визуально меняться без отдельной задачи;
- текущие route остаются рабочими;
- `HashRouter` не меняется без отдельного решения;
- живые страницы не заменяются заглушками;
- лидерские маршруты не схлопываются до одного `/leader`.

## Что уже готово

- Зафиксирована карта текущих маршрутов и ограничений миграции: `docs/architecture-routing-inventory.md`.
- Добавлен `navigationConfig` с реальными route и binding к текущим компонентам.
- Добавлены роли, feature flag metadata и helpers для фильтрации навигации.
- Добавлена Command Palette поверх текущего интерфейса.
- Добавлен `RouteErrorBoundary`.
- Добавлен первый layout-шаг: `AppWorkspaceFrame`.

## Что пока НЕ сделано

- Sidebar участника ещё не использует `navigationConfig` как источник данных.
- Sidebar лидера ещё не использует `navigationConfig` как источник данных.
- Mobile bottom nav ещё ручной.
- Роль пользователя пока mock в navigation context.
- Feature flags пока mock.
- Header ещё находится внутри `App.tsx`.
- Выбор страниц и leader tabs ещё находится внутри `App.tsx`.
- Жёстких route guards пока нет, и это намеренно.

## Шаг 3.0 — стабилизация текущего слоя

### Цель

Убедиться, что уже добавленные слои не ломают текущий продукт.

### Что проверить

- `/`
- `/my-path`
- `/community`
- `/connections`
- `/leader`
- `/leader/entry`
- `/leader/connections`
- Command Palette через кнопку поиска
- Command Palette через `Ctrl/⌘ + K`
- Закрытие Command Palette через `Escape` и клик вне окна

### Definition of Done

- `npm run lint` проходит.
- `npm run build` проходит.
- Визуальный интерфейс не изменён непреднамеренно.
- Ключевые route открываются.

## Шаг 3.1 — подключить `navigationConfig` к sidebar участника

### Цель

Сделать `navigationConfig` первым реальным источником данных для участнической навигации.

### Что меняем

- Только desktop sidebar участника.
- Старый массив пунктов участнического меню заменяется данными из `navigationConfig`.
- Внешний вид, порядок и активное состояние должны остаться прежними.

### Что НЕ трогаем

- Leader sidebar.
- Mobile bottom nav.
- Router.
- Page switch.
- Header.
- Auth/guards.

### Риск

Низкий, если не менять стили и route.

### Definition of Done

- Участнический sidebar выглядит как раньше.
- Пункты ведут на те же URL.
- Активное состояние сохраняется.
- Command Palette и sidebar используют один источник route metadata.

---

## Шаг 3.2 — подключить `navigationConfig` к sidebar лидера

### Цель

Убрать ручной массив вкладок лидерской консоли и описывать лидерскую навигацию через общий config.

### Что меняем

- Только leader sidebar.
- Route `/leader`, `/leader/entry`, `/leader/requests`, `/leader/connections`, `/leader/contribution`, `/leader/monetization`, `/leader/settings` сохраняются.
- Внешний вид и badges сохраняются.

### Что НЕ трогаем

- Логику страниц лидера.
- Switch выбора leader tab.
- Router.
- Guards.

### Риск

Средний, потому что лидерская консоль критична. Делать отдельным PR после успешного шага 3.1.

### Definition of Done

- Все лидерские вкладки открываются по прежним URL.
- Активная вкладка определяется корректно.
- Лидерская консоль не схлопнулась до одного route.

## Шаг 3.3 — подключить `navigationConfig` к mobile bottom nav

### Цель

Синхронизировать мобильную навигацию с desktop navigation и Command Palette.

### Что меняем

- Только mobile bottom nav.
- Сохраняем текущий набор быстрых пунктов и визуальный стиль.

### Что НЕ трогаем

- Desktop sidebar.
- Header.
- Router.
- Page switch.

### Риск

Средний: мобильный UX легко испортить, поэтому проверять отдельно.

### Definition of Done

- На мобильной ширине пункты выглядят как раньше.
- Активное состояние работает.
- Переходы открывают прежние route.

## Шаг 3.4 — заменить mock роли и flags на централизованный контекст

### Цель

Перестать держать role/feature flags как локальный mock внутри navigation hook.

### Что меняем

- Добавляем безопасный источник текущей роли пользователя.
- Добавляем безопасный источник feature flags.
- Пока используем их только для видимости navigation items.

### Что НЕ трогаем

- Прямой доступ к route.
- Protected routes.
- Role guards.
- Auth flow.

### Риск

Средний. Нельзя сразу закрывать route, пока не согласована модель доступа.

### Definition of Done

- Лидер видит лидерские пункты.
- Участник не видит лидерские пункты в навигации.
- Прямые route не ломаются без отдельного решения.

## Шаг 3.5 — вынести Header из `App.tsx`

### Цель

Снизить размер `App.tsx`, не меняя визуальный header.

### Что меняем

- Создаём компонент header.
- Передаём в него только необходимые props/state callbacks.
- Сохраняем Command Palette trigger, theme toggle, notifications, avatar menu и community dropdown.

### Что НЕ трогаем

- Sidebar.
- Page switch.
- Router.
- Логику страниц.

### Риск

Средний: в header много состояния и dropdown-логики.

### Definition of Done

- Header визуально совпадает с текущим.
- Dropdown сообществ работает.
- Avatar menu работает.
- Theme toggle работает.
- Command Palette открывается как раньше.

## Шаг 3.6 — вынести sidebar-компоненты

### Цель

Разделить участническую и лидерскую навигацию на отдельные компоненты.

### Порядок

1. `ParticipantSidebar`.
2. `LeaderSidebar`.
3. Общие маленькие элементы, если появится повторение.

### Что НЕ трогаем

- Router.
- Page switch.
- Содержимое страниц.

### Риск

Средний. Делать после шагов 3.1 и 3.2, когда данные уже идут из `navigationConfig`.

### Definition of Done

- Sidebar визуально не изменился.
- Активные состояния работают.
- Route сохраняются.

## Шаг 3.7 — вынести route surface / page switch

### Цель

Вынести из `App.tsx` выбор текущей страницы и leader tab в отдельный компонент.

### Что меняем

- Создаём компонент для отображения текущей рабочей поверхности.
- Сохраняем существующие lazy-loaded страницы.
- Сохраняем текущие props-based route bindings до отдельного решения по router.

### Что НЕ трогаем

- `HashRouter`.
- Список route в `src/main.tsx`.
- Сами страницы.

### Риск

Высокий. Делать только после стабилизации sidebar/header.

### Definition of Done

- Все текущие страницы открываются.
- Все leader tabs открываются.
- Никакая страница не заменена заглушкой.

## Шаг 3.8 — обсудить route guards

### Цель

Понять, когда можно безопасно закрывать route по роли, а не только скрывать пункты меню.

### Что нужно решить до реализации

- Какой источник auth считается правдой.
- Что делать с прямым переходом участника на лидерский URL.
- Нужна ли страница “нет доступа” или мягкий redirect.
- Как не сломать демо/презентационные сценарии.

### Риск

Высокий. Не делать до согласования.

## Шаг 3.9 — только после этого обсуждать новый router/layout

### Цель

Перейти к более чистой структуре layouts/routes, когда App уже разгружен.

### Условия старта

- Navigation config реально используется в desktop и mobile navigation.
- Header вынесен.
- Sidebar вынесен.
- Page switch вынесен.
- Роли/flags подключены к видимости navigation.
- Все route из текущей карты сохранены.

### Что нельзя делать резко

- Менять `HashRouter` на `BrowserRouter`.
- Делать `App.tsx` минимальным одним PR.
- Схлопывать leader routes.
- Создавать заглушки вместо живых страниц.
